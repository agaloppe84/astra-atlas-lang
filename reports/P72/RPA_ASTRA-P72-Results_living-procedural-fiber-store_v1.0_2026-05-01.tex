\begin{abstract}
Ce rapport fige les r\'esultats P72 du repo \Code{astra-atlas-lang}. P72
introduit un store procedural fibre vivant: etat froid persistant, etat runtime
temporaire, actions read/query/update/delete/audit, compaction, close, reopen
et replay du journal. Le sprint valide le chemin fonctionnel close/reopen, mais
garde une decision scientifique conservatrice car le modele de cout vivant
reste en drift dur.
\end{abstract}

\section{But de ASTRA-P72}

P72 teste ASTRA comme stockage procedural vivant. ASTRA n'est pas un disque
virtuel qui contient toutes les fibres generees. ASTRA persiste un etat froid
procedural suffisant pour regenerer les fibres utiles a chaud. Le cache runtime
peut disparaitre si la reouverture restaure le meme etat logique observable.

\section{Modele cold/runtime/reopen}

L'etat froid contient contrat, generateurs, parametres quantifies,
dictionnaires, index, residus, journal, checkpoints, manifeste, checksums,
metadata d'audit, metadata de surete et politique acteur.

L'etat runtime contient fibres materialisees, cache chaud, etats acteurs,
queues d'action, index temporaires et vues decodees. Ces structures sont un
working set mesure, pas une source de verite cachee.

\section{Decision}

Decision P72:
\Decision{RECALIBRATE_P72_LIVING_COST_MODEL}.

P72 etablit que le cycle vivant fonctionne: exact roundtrip, retrieval, journal
replay, compaction, guard et reopen equivalence passent localement. P72
n'etablit pas encore un modele de cout vivant final.

\section{Ce que P72 etablit}

\begin{itemize}
  \item un store vivant peut etre encode, ouvert, modifie, compacte, ferme et
  rouvert;
  \item l'equivalence logique observable survit au reopen;
  \item le cache runtime n'est pas une memoire cachee necessaire a la
  correction;
  \item le cout persistant et le cout runtime sont mesures separement;
  \item la garde incompressible reste honnete.
\end{itemize}

\section{Ce que P72 ne prouve pas}

\begin{itemize}
  \item pas de validation scientifique multi-machine;
  \item pas de long historique de reopen/replay;
  \item pas de corruption/recovery generalise;
  \item pas de modele final de cout, car le drift persistant reste dur;
  \item pas de garantie sur datasets externes non testes.
\end{itemize}

\section{Recommendation P73}

P73 devrait tester des sessions vivantes plus longues: plusieurs cycles
open/close, rafales d'updates, injection de corruption, recovery explicite,
bornes de replay journal, et calibration du cout persistant declare pour
reduire le drift sous seuil.
