\section{But de ASTRA-P69}

P69 formalise ce qui est stocke lorsque l'architecture
\Code{address_fiber_actor_managed_v1}, promue en P68, est utilisee. L'espace
virtuel n'est pas materialise globalement. Ce qui est stocke est un code
procedural fini: espace d'adresses, schema de fibre, generateur, parametres,
dictionnaire/ROM, index, residus, journal, politique acteur, budgets, gates de
s\^urete et metadata d'audit.

P68 a promu le candidat address-fiber actor-managed:

\begin{itemize}
  \item standard gate: \Code{PASS};
  \item ambitious gate: \Code{PASS};

\section{Question centrale: forme stockee du code procedural}

La question P69 est:

\begin{quote}
Quelle forme finie est reellement payee, et comment empecher qu'un cache, un
journal, un index, un acteur ou une metadata soit cache hors du cout?
\end{quote}

Le modele retenu est:

\begin{verbatim}
ProceduralCode =
  AddressSpace
  + FiberSchema
  + Generator
  + QuantizedParameters
  + Index
  + Residuals
  + Journal
  + ActorPolicy
  + Budgets
  + SafetyGates
  + AuditMetadata
\end{verbatim}

\section{Parser et typechecker}

Le module \Code{src/p69.rs} ajoute un parser/typechecker specialise. Il verifie:

\begin{itemize}
  \item l'architecture \Code{address_fiber_actor_managed_v1};
  \item le modele de cout \Code{measured_or_declared_contract_v1};
  \item les references address space, fiber schema, generator et actor policy;
  \item \Code{budget_bytes > 0};
  \item les politiques projection, journal, audit, compaction et cache connues;
  \item la coherence journal/audit/compaction entre fibre et acteur;
  \item tous les champs de stockage requis marques \Code{accounted};
  \item \Code{all_storage_counted = true};
  \item zero conflicts et zero stale reads.
\end{itemize}

\section{Ce que P69 etablit}

\begin{itemize}
  \item une forme stockee explicite du code procedural ASTRA;
  \item un lien contractuel avec l'architecture promue P68;
  \item une decomposition de cout qui inclut cache, journal, acteur, index,
    residus, audit, manifest et safety metadata;
  \item des invalides dedies pour empecher le drift du contrat;
  \item des exports locaux ignores par Git.
\end{itemize}

\section{Ce que P69 ne prouve pas encore}

\begin{itemize}
  \item pas de validation scientifique finale multi-machine;
  \item pas de dataset externe lourd;
  \item pas encore de comparaison declared-vs-measured sur plusieurs familles de
    contrats;
  \item pas de garantie que tous les futurs contrats seront optimaux.
\end{itemize}

\section{Recommandation P70}

P70 devrait introduire des campagnes de replay liees au contrat:

\begin{itemize}
  \item rejouer le contrat P69 sur plusieurs fixtures;
  \item comparer les bytes declares et les bytes mesures;
  \item ajouter la detection de drift contractuel;
  \item garder les mutants P69 comme gates de non-regression;
  \item preparer un protocole multi-machine.
\end{itemize}

\section{Conclusion}

P69 repond a la question "qu'est-ce qui est stocke?" par un contrat
procedural fini, typechecke et auditable. L'espace virtuel reste non
materialise globalement; les couts reels du code procedural sont decomposes et
comptes. La prochaine etape est de rejouer ce contrat dans des campagnes P70.
