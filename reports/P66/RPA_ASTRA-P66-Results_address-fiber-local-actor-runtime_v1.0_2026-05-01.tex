\section{But de ASTRA-P66}

P66 formalise le mod\`ele \Code{address-fiber}. L'espace virtuel est une base
d'adresses; la donn\'ee utile est une fibre locale attach\'ee \`a une adresse.

\[
  x \in \Omega_{\mathrm{virtual}}, \qquad
  F_x = \text{fibre locale attach\'ee \`a } x
\]

\[
  Eval(c,x) = \text{g\'en\'eration contr\^ol\'ee de } F_x
  \text{ ou de } F_{N(x,r)}
\]

Le runtime doit g\'en\'erer la fibre utile, ou un petit voisinage fibr\'e, sans
mat\'erialiser l'espace global.

\section{Position apr\`es P65-2}

P65-2 a montr\'e que \Code{single_local_actor} peut obtenir des gains nets
forts avec overhead calibr\'e sous 15\% sur certaines configurations. Les
meilleures configurations P65-2 \'etaient:

\begin{longtable}{@{}p{0.18\linewidth}p{0.45\linewidth}rrr@{}}

\bottomrule

P66 reprend cette direction, mais change le niveau d'analyse: l'acteur ne g\`ere
plus seulement un voisinage abstrait, il g\`ere une fibre locale ou un voisinage
fibr\'e.

\section{Hypoth\`ese centrale: point d'adresse plus fibre locale}

\begin{quote}
L'adresse est un point dans l'espace virtuel. La donn\'ee utile est une fibre
locale au-dessus de ce point. Le runtime doit g\'en\'erer la fibre utile, ou un
voisinage fibr\'e, sans mat\'erialiser l'espace global.
\end{quote}

\section{Mod\`ele AddressFiber}

Le mod\`ele \Code{AddressFiber} expose:

\begin{itemize}
  \item identifiant d'adresse et coordonn\'ee de base;
  \item type de fibre;
  \item unit\'es d\'eclar\'ees, atteignables, lisibles, modifiables, s\^ures,
        effectives et g\'en\'er\'ees;
  \item octets payload, index, cache, journal, audit et metadata;
  \item liaison optionnelle \`a un acteur local;
  \item statut de s\^uret\'e et raisons de d\'ecision.
\end{itemize}

Toute m\'emoire fibre/acteur est compt\'ee comme co\^ut r\'eel.

\section{D\'ecision}

D\'ecision repo:

\[
  \Decision{VALIDER_IMPL_P66_ADDRESS_FIBER_RUNTIME}
\]

D\'ecision scientifique:

\[
  \Decision{RECALIBRATE_P66_ADDRESS_FIBER_MODEL}
\]

\Code{PROMOTE_P66_ADDRESS_FIBER_ARCHITECTURE} n'est pas atteint, car
l'overhead acteur-fibre reste au-dessus du seuil candidat.

\section{Ce que P66 \'etablit}

\begin{itemize}
  \item Le mod\`ele address-fiber est impl\'ement\'e et test\'e.
  \item \Code{actor_managed_fiber} domine les strat\'egies fibre sur les quatre
        workloads.
  \item Les m\'etriques CRUD/audit/compaction sont compt\'ees.
  \item Les conflits et stale reads restent \`a 0 sur les meilleures entr\'ees.
\end{itemize}

\section{Ce que P66 ne prouve pas encore}

\begin{itemize}
  \item P66 ne valide pas scientifiquement l'architecture finale.
  \item Les workloads restent internes et d\'eterministes.
  \item Il n'y a pas de dataset externe ni de r\'ep\'etition multi-machine.
  \item L'overhead acteur-fibre reste trop haut.
\end{itemize}

\section{Recommandation P67}

P67 devrait porter sur:

\begin{itemize}
  \item r\'eduction de l'overhead \Code{actor_managed_fiber};
  \item reprise de la grille de calibration P65-2 au niveau fibre;
  \item gates standard+ambitious formels;
  \item stress cache/journal/compaction;
  \item r\'ep\'etition multi-machine avant promotion scientifique.
\end{itemize}

\section{Conclusion}

P66 est un jalon conceptuel et syst\`eme important: l'adresse devient un point,
la donn\'ee utile devient une fibre locale, et le runtime mesure le co\^ut de
g\'en\'erer et g\'erer cette fibre. Le meilleur signal est
\Code{actor_managed_fiber}; la limite bloquante reste l'overhead.
