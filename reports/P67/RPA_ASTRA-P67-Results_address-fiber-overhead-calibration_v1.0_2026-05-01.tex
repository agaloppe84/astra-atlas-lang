\section{Objectif P67}

P67 calibre l'overhead du mod\`ele \Code{actor_managed_fiber} introduit en P66.
La question est la suivante: peut-on garder un fort
\Code{address_fiber_net_gain} tout en ramenant
\Code{avg_actor_overhead_ratio} sous 0.15?

Les co\^uts de cache, journal, audit, metadata et compaction restent compt\'es.
Aucun timing n'est goldenis\'e.

\section{Position apr\`es P66}

P66 a confirm\'e la direction \Code{address-fiber}: l'adresse est un point dans
l'espace virtuel, et la donn\'ee utile est une fibre locale au-dessus de ce
point. La meilleure strat\'egie observ\'ee \'etait \Code{actor_managed_fiber}.

Limite P66:

\begin{itemize}
  \item overhead standard moyen: 0.294461;
  \item overhead ambitious moyen: 0.249963;
  \item d\'ecision: \Decision{RECALIBRATE_P66_ADDRESS_FIBER_MODEL}.
\end{itemize}

\section{Param\`etres test\'es}

P67-v1 explore:

\begin{itemize}
  \item radius: 1, 2, 3, 5 en standard; 2, 3, 5 en ambitious;
  \item budget bytes: 524288 \`a 4194304;
  \item cache policy: \Code{on}, \Code{compact};
  \item journal policy: \Code{lazy}, \Code{compact};
  \item audit policy: \Code{minimal}, \Code{sampled};
  \item compaction policy: \Code{threshold}, \Code{aggressive};
  \item query locality: \Code{clustered}, \Code{mixed};
  \item fiber projection: \Code{shallow}, \Code{medium}.
\end{itemize}

P67-v1 garde \Code{update_rate} et \Code{metadata_policy} comme dimensions
d\'eterministes standard.

\section{Pareto front}

Les deux campagnes produisent une fronti\`ere de Pareto non vide:

\begin{itemize}
  \item standard: 32 entr\'ees Pareto retenues;
  \item ambitious: 32 entr\'ees Pareto retenues.
\end{itemize}

La zone candidate favorise cache compact, journal compact, audit minimal,
localit\'e clustered, projection shallow et compaction threshold/aggressive
selon la pression de requ\^etes.

\section{Crit\`eres de promotion}

Les crit\`eres num\'eriques candidats sont satisfaits par les meilleurs
\Code{best_balanced} locaux:

\begin{itemize}
  \item standard: net gain $>$ 3.0 et overhead $<$ 0.15;
  \item ambitious: net gain $>$ 2.5 et overhead $<$ 0.18;
  \item conflicts = 0;
  \item stale reads = 0;
  \item budget refusals = 0;
  \item update/audit/compaction sont non nuls et compt\'es.
\end{itemize}

Cependant, P67-v1 ne retourne pas \Decision{PROMOTE_P67_ADDRESS_FIBER_ARCHITECTURE}.
La promotion doit \^etre formalis\'ee en P68 comme gate pair\'e standard +
ambitious, avec dimensions update/metadata explicites.

\section{Limites}

\begin{itemize}
  \item workloads internes d\'eterministes;
  \item pas de dataset externe;
  \item pas de multi-machine;
  \item update rate et metadata policy restent fixes en P67-v1;
  \item les ratios \'elev\'es sont des sorties de calibration, pas une validation scientifique finale;
  \item aucun timing golden.
\end{itemize}

\section{Recommandation P68}

P68 doit formaliser un gate pair\'e standard + ambitious:

\begin{itemize}
  \item combiner les deux rapports dans un \'evaluateur de promotion;
  \item rendre \Code{update_rate} et \Code{metadata_policy} explicitement calibrables;
  \item rejouer sur une seconde machine si possible;
  \item conserver tous les co\^uts fibre/acteur;
  \item ne promouvoir qu'avec preuve locale structur\'ee et limites explicites.
\end{itemize}
